# SSO Cookie-Domain E2E Test

Reproduces the SSO handoff cookie-domain regression that broke
self-hosted deployments on any host other than `*.kodus.io`
(e.g. Dmitry's `*.web.scorpion.co`). Sets up Keycloak as a SAML IdP
and Kodus as the SP, both on `*.kodus.lvh.me`, behind a TLS-terminating
Caddy reverse proxy. Logs in as a SAML user end-to-end and verifies
the `Set-Cookie` Domain attribute in the browser.

## What this validates

The fix in `apps/api/src/controllers/auth.controller.ts` and
`apps/api/src/utils/derive-sso-cookie-domain.ts` derives the cookie
`Domain` as the smallest common DNS suffix between:

- `req.get('host')` — the API host emitting `Set-Cookie`
- `process.env.API_FRONTEND_URL` — the frontend host that needs to
  read the cookie on `/sso-callback`

Unit tests cover the derivation function exhaustively
(`apps/api/src/utils/__tests__/derive-sso-cookie-domain.spec.ts`)
and the controller wiring
(`apps/api/src/controllers/__tests__/auth.controller.sso-cookie.spec.ts`).
This E2E test exists to verify the **full SAML round-trip**: that
Keycloak → API callback → browser actually delivers a cookie with the
expected `Domain=` and the `/sso-callback` page consumes it cleanly.

## Prerequisites

1. Dev backing services running (postgres, mongo, rabbit):
    ```sh
    yarn docker:start
    ```
2. Production images of API + Web built locally:
    ```sh
    WEB_TAGS=kodus-web:sso-e2e RELEASE_VERSION=sso-e2e \
        docker buildx bake -f docker-bake.hcl web
    API_TAGS=kodus-api:sso-e2e API_CLOUD_MODE=false \
        docker buildx bake -f docker-bake.hcl api
    ```
3. mkcert installed and trusted on the host:
    ```sh
    brew install mkcert
    sudo mkcert -install
    ```
   The root CA install is what makes the browser trust the wildcard
   cert generated under `.tmp/sso-e2e-tls/`. Skip it and Chrome shows
   "Not secure" + may refuse to store the `secure: true` cookie.
4. `python3` available on PATH (the bootstrap scripts use it for JSON
   parsing — only `json`/`os`/`sys` from the stdlib).

## Topology

`lvh.me` is a public DNS zone whose every subdomain resolves to
`127.0.0.1` — no `/etc/hosts` editing needed. Caddy on port 443
terminates TLS using the mkcert wildcard cert.

```
Browser ──► https://app.kodus.lvh.me   (Caddy → kodus-web:3000)
Browser ──► https://api.kodus.lvh.me   (Caddy → kodus-api:3001)
Browser ──► http://localhost:8080      (Keycloak)
```

The expected cookie `Domain` is `.kodus.lvh.me`.

## Running

### Quick — no-browser regression check

Runs unit + integration tests + a 2-shape runtime smoke (cloud and
self-hosted simultaneously). No mkcert, no Keycloak, no manual steps.
~60 seconds end-to-end the first time, ~10 seconds on subsequent runs
(the prod web image is cached).

```sh
./scripts/sso-e2e/test-all.sh
```

### Full — with browser SAML round-trip

Adds a 4th layer: real Keycloak + Caddy + browser, exercising the
SAML callback in production-mode TLS. Prerequisites listed in the
"Prerequisites" section above (mkcert + sudo install).

```sh
./scripts/sso-e2e/test-all.sh --e2e
```

### Switching between SaaS-shape and Dmitry-shape

The browser layer parameterizes the topology via `SSO_E2E_DOMAIN`:

```sh
# SaaS shape (default) — common parent ".kodus.lvh.me" (3 labels, analog of .kodus.io)
./scripts/sso-e2e/run.sh

# Dmitry shape — common parent ".web.scorpion.lvh.me" (4 labels, analog of .web.scorpion.co)
SSO_E2E_DOMAIN=web.scorpion.lvh.me ./scripts/sso-e2e/run.sh
```

The cert generated by mkcert covers both wildcards in the same SAN
list, so you can switch shapes without regenerating it. Caddy serves
both name patterns with the same TLS material.

### Just the browser layer

If you specifically want to debug the SAML round-trip and skip the
unit tests, call the orchestrator directly:

```sh
./scripts/sso-e2e/run.sh
```

The script:

1. Sanitizes `.env` to a docker-compatible `--env-file`
   (`.tmp/sso-e2e-api.env`).
2. Generates the wildcard TLS cert via mkcert if missing.
3. Boots Caddy + Keycloak + API + Web via
   `docker/sso-e2e/docker-compose.yml`.
4. Configures Keycloak (realm `kodus-sso-e2e`, SAML client
   `kodus-orchestrator`, user `sso-user@kodus-test.com` /
   `TestSso!2026`).
5. Configures Kodus (signup an admin, create org, register the SAML
   IdP via `POST /sso-config`).
6. Re-runs Keycloak setup so the SAML client's ACS URL is bound to
   the new orgId.
7. Prints the manual verification steps.

## Verification

Follow the steps printed by `run.sh`:

1. Open `https://api.kodus.lvh.me/auth/sso/login/<orgId>` (printed
   by the script). This is a direct call to the API that bypasses
   the front-end "active" gate (the seeded SSO config is
   `active: false` so the front-end's gate would otherwise refuse
   to redirect — the cookie-domain code path on the callback is
   identical either way).
2. Sign in on Keycloak: `sso-user@kodus-test.com` / `TestSso!2026`.
3. Land on `https://app.kodus.lvh.me/sso-callback` and then the app.
4. Open Chrome DevTools → Application → Cookies →
   `https://app.kodus.lvh.me`. Confirm:

    | Field | Value |
    |---|---|
    | Name | `sso_handoff` (briefly — 15s lifetime) |
    | **Domain** | **`.kodus.lvh.me`** |
    | Path | `/` |
    | Secure | ✓ |
    | SameSite | `Lax` |

The `Domain=.kodus.lvh.me` is the proof. Before the fix it was
hard-coded to `.kodus.io`, so any self-hosted topology
(Dmitry's `*.web.scorpion.co`, customer-A's `*.example.com`,
internal `*.intranet.local`, …) would have its cookie scoped to
a domain the frontend's host doesn't belong to. The browser
silently drops it and SSO handoff fails with no obvious error.

If you can't open DevTools (CI / headless), instrument the API
controller temporarily — see "Method A" in the appendix below.

## Appendix — Method A: instrumented log

If the browser path isn't available (CI, no GUI, headless run), add
one line to `apps/api/src/controllers/auth.controller.ts` just above
`res.cookie('sso_handoff', ...)`:

```ts
console.log(
    `[SSO_E2E] req.host=${req.get('host')} frontendUrl=${frontendUrl} nodeEnv=${process.env.API_NODE_ENV} → cookieDomain=${cookieDomain}`,
);
```

Rebuild, restart `kodus-api`, run the SAML flow once, then:

```sh
docker logs kodus-sso-e2e-api 2>&1 | grep SSO_E2E
```

Expected output:

```
[SSO_E2E] req.host=api.kodus.lvh.me frontendUrl=https://app.kodus.lvh.me nodeEnv=production → cookieDomain=.kodus.lvh.me
```

Remove the `console.log` before committing.
