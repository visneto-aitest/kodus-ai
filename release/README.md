# `release/` — Feature catalog and release artifacts

This directory holds artifacts that drive the feature lifecycle and the
public changelog.

## Files

- **`features.yaml`** — the catalog. Single source of truth for every
  user-facing feature's lifecycle stage, audience, and copy. Edited by hand
  through PRs; promotion of a feature requires CODEOWNERS approval.
- **`features-snapshot.json`** — read model derived from `features.yaml`.
  Regenerated by `scripts/feature-gate/generate-snapshot.ts`. Read by
  `FeatureGateService` at runtime and by the changelog pipeline. Embedded in
  Docker images for self-hosted deployments.
- **`features-snapshot.schema.json`** — JSON Schema for the snapshot. Used by
  editors for validation.

## How the system works

```
release/features.yaml   ←  edited via PR, CODEOWNERS approves promotions
        │
        ▼
yarn feature-gate:snapshot
        │
        ▼
release/features-snapshot.json   ←  committed, shipped in Docker images
        │
        ▼
FeatureGateService.isEnabled(...) at runtime
   │
   ├─ Cloud: stage gate -> audience gate -> PostHog flag (org/repo conditions)
   └─ Self-hosted: stage gate -> audience gate -> BETA_FEATURES env
```

The PostHog feature flags themselves stay in PostHog (each `key` in the
catalog corresponds to a flag of the same name there). They handle real-time
operational rollout — percentage, per-org allowlists, kill-switch. The
catalog handles lifecycle and customer-facing communication.

## Stage semantics

| Stage                  | Public-changelog visible? | Default audience (cloud)                 | Self-hosted                       |
| ---------------------- | ------------------------- | ---------------------------------------- | --------------------------------- |
| `alpha`                | no                        | orgs on `release_track = alpha`          | nobody                            |
| `beta`                 | yes (with `[Beta]` badge) | orgs on `release_track in (beta, alpha)` | only when `BETA_FEATURES=true`    |
| `general-availability` | yes                       | everyone                                 | everyone                          |

## Promoting a feature

1. Open a PR editing `release/features.yaml` with the new `stage`.
2. Run `yarn feature-gate:snapshot` and commit the resulting
   `features-snapshot.json`.
3. CI verifies the snapshot is in sync via `yarn feature-gate:snapshot:check`.
4. CODEOWNERS approval is required for any stage change.
5. On merge, the changelog pipeline (Fase 3) drafts an entry, posts to
   Discord, and updates Mintlify when the new stage is `beta` or
   `general-availability`.

## How PRs flow into the changelog

PR title prefix (Conventional Commits) decides changelog routing:

- `feat:` -> Improvements section.
- `fix:` -> Bug fixes section.
- `perf:` -> Performance section.
- `docs:`, `style:`, `refactor:`, `test:`, `chore:`, `ci:`, `build:` -> hidden.

There are no GitHub labels and engineers don't link PRs to features manually.
When a feature is promoted, the LLM uses `name + description` from
`features.yaml` to draft the changelog entry. CI enforces the PR title via
`.github/workflows/pr-title-check.yml`.

## Changelog automation (Fase 3)

There's **one** changelog — same Mintlify entries, same Discord channel,
same audience for cloud and self-hosted users. The two deploy trains ship
the same features within a few days of each other, so we publish once
and call it the release.

### Why the trigger is `selfhosted-*`

We use the self-hosted tag as the trigger purely to avoid duplicate posts
— we'd get one cloud and one self-hosted entry for the same promotion
otherwise. The release entry is **not labeled "self-hosted"**: the
Mintlify title and Discord message read `Release X.Y.Z`, and the version
in the user's hands matches.

Editing `features.yaml` (promoting a stage, adding a description) doesn't
publish anything by itself — entries land in the changelog only when the
next `selfhosted-X.Y.Z` tag is cut.

### Pipeline

`.github/workflows/changelog-publish.yml` runs on push of any
`selfhosted-X.Y.Z` tag. It finds the previous `selfhosted-X.Y.Z` tag,
diffs `release/features.yaml` between the two, and collects every
promotion that hit `beta` or `general-availability` in that range. Alpha
promotions are skipped — those are coordinated directly with design
partners.

### Output

One MDX entry per release, titled `Release X.Y.Z`. Committed to
`docs/changelog/YYYY-MM-DD-selfhosted-X-Y-Z.mdx` by
`kodus-changelog-bot`. The same content is posted to
`DISCORD_WEBHOOK_SELFHOSTED` (the channel cloud and self-hosted
customers both follow).

### Audience-specific entries

Each promoted feature carries an `audience` value from
`features.yaml` (`[cloud]`, `[self-hosted]`, or both). The publish
script renders an audience suffix on the entry headline so readers
can spot deploy-specific changes:

- `[Beta]` — both deployments (default)
- `[Beta · Cloud only]` — only cloud customers see this
- `[GA · Self-hosted only]` — only self-hosted Docker users see this

Default for catalog entries without an explicit `audience` field is
"both" — no extra badge.

### LLM draft

`scripts/feature-gate/publish-changelog.ts` calls Gemini (default
`gemini-3-flash-preview`, override with the `GEMINI_MODEL` workflow var)
with each promotion's catalog entry, `release/voice.md`, and the file
paths the feature touches in the repo. Asks for `{ headline, body }`
JSON via Gemini's structured-output schema.

### Required GitHub secrets

| Secret                       | Purpose                                                  |
| ---------------------------- | -------------------------------------------------------- |
| `GEMINI_API_KEY`             | Google AI API key for headline + body generation.        |
| `DISCORD_WEBHOOK_SELFHOSTED` | Self-hosted release channel (existing, reused).          |

Optional repo `vars`:

| Var            | Default                  | Purpose                                          |
| -------------- | ------------------------ | ------------------------------------------------ |
| `GEMINI_MODEL` | `gemini-3-flash-preview` | Override the model when a stable / newer rev is available. |

When a secret is missing the script degrades gracefully (LLM falls back to
deterministic copy from `description`; Discord posts are skipped with a
warning). The MDX is still written so the catalog change is auditable.

### Setting an organization's release track

Kodus staff control tracks via script (no admin UI). The script updates the
DB and mirrors the new value to PostHog group properties in one shot.

Tracks are cumulative:

| Track    | Sees features at stage |
| -------- | ---------------------- |
| `stable` | only `ga`              |
| `beta`   | `ga` + `beta`          |
| `alpha`  | `ga` + `beta` + `alpha` |

```sh
yarn feature-gate:set-track --org-id=<uuid> --track=stable
yarn feature-gate:set-track --org-id=<uuid> --track=beta --dry-run
yarn feature-gate:set-track --org-id=<uuid> --track=alpha --env=.env.prod
```

Default for every new org is `beta`. Stability customers get flipped to
`stable`. Design partners and the Kodus dogfood org get `alpha`. PostHog
flag conditions remain available for operational fine-tuning (rollout %,
kill-switch, per-org overrides) on top of the track gate.

### Dry run locally

```sh
yarn feature-gate:changelog:dry --tag selfhosted-2.0.20
```

### Tone of voice

Edit `release/voice.md`. The LLM consumes the file verbatim — keep it short
and concrete. Changes go through PR review like any other source-of-truth.

## Editing rules

- `features.yaml` is hand-edited through PRs. CODEOWNERS approval required.
- Don't hand-edit `features-snapshot.json` — it's regenerated. CI fails
  otherwise.
- A feature key must match a real PostHog feature flag key (kebab-case).
- Don't set `private_beta_orgs` on stages other than `alpha` or `beta`; the
  validator will reject.
