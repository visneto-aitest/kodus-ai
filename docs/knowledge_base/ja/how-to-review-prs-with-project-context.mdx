title: "プロジェクト固有のコンテキストでPRをレビューする方法"
description: "Kodyにプロジェクトドキュメント、ハンドブック、アーキテクチャガイドへのアクセスを提供して、より関連性の高いコードレビューを行います。"
---

汎用のコードレビューは汎用的な問題を検出します。プロジェクトに実際に関連するフィードバックを得るには、Kodyがあなたの特定のコンテキスト — アーキテクチャの決定、規約、ビジネスルール — を理解する必要があります。

## コンテキストを提供する方法

### メモリ

会話を通じてKodyに規約を教えます：

```
@kody remember: このプロジェクトは六角形アーキテクチャを使用します。
ドメイン層はインフラストラクチャに依存してはいけません。
```

メモリは永続化され、将来のすべてのレビューに適用されます。

### ルールにおけるファイル参照

ルール内で直接ドキュメントを参照します：

```
指示: 新しいエンドポイントが@file:docs/api-conventions.mdのパターンに従い、
@file:src/shared/base-controller.tsで定義されたベースクラスを
使用していることを検証します。
```

### カスタムプロンプト

Kodyがすべてのレビューに含めるプロジェクト固有の指示を追加します：

1. **コードレビュー設定** → **カスタムプロンプト** に移動します
2. アーキテクチャの説明、チームガイドライン、ドメイン知識などのコンテキストを追加します
3. Kodyはすべてのコードレビュー分析にこれを含めます

### MCPプラグイン

Kodyにリッチなコンテキストへのアクセスを提供するために外部ツールを接続します：

- **Jira/Linear** — タスク要件と受け入れ基準
- **Slack** — チームの議論と決定
- **Google Docs** — 仕様と設計ドキュメント
- **カスタムMCPサーバー** — チームが使用するあらゆるツール

## ワークフロー例

1. **メモリ**でアーキテクチャルールを定義します：`@kody remember: このサービスでCQRSパターンに従います`
2. ドキュメントを参照する**レビュールール**を作成します：`@file:docs/architecture.md`
3. **Jira**を接続してKodyがPRをタスク要件に対して検証するようにします
4. ドメイン固有の知識のために**カスタムプロンプト**を追加します

詳細については、[カスタムプロンプト](/how_to_use/ja/code_review/configs/custom_prompts)と[プラグイン](/how_to_use/ja/code_review/plugins)を参照してください。
