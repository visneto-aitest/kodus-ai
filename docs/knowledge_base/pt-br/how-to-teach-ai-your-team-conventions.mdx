Todo time tem contexto não escrito — preferências tecnológicas, convenções de nomenclatura, decisões arquiteturais, detalhes de migração. Em vez de documentar tudo isso antecipadamente, você pode ensinar o Kody por meio de conversas naturais para que futuras revisões sejam mais relevantes e sugestões repetitivas e ruins desapareçam.

## Como funciona

Quando você fala com o Kody nos comentários de PR, ele detecta quando você está declarando contexto do codebase ou uma preferência do time e salva isso automaticamente como uma **Memória**. As Memórias são então aplicadas como contexto de alta prioridade em todas as futuras revisões de código e conversas.

### Ensino explícito

Diga diretamente ao Kody para lembrar algo:

```
@kody remember: este repositório espelha uma API de terceiros, então alguns campos de payload permanecem intencionalmente em snake_case.
```

```
@kody please remember: este serviço segue arquitetura hexagonal e mantém adapters nas bordas.
```

### Aprendizado implícito

Expresse uma preferência naturalmente — o Kody a capta:

```
@kody este repositório espelha uma API de terceiros, então alguns campos de payload permanecem intencionalmente em snake_case.
```

```
@kody neste serviço, os testes normalmente ficam ao lado dos arquivos de implementação, não em uma pasta central de testes.
```

```
@kody o módulo de billing está no meio de uma migração, então prefira mudanças incrementais a refactors amplos.
```

## O que o Kody não salva

O Kody é seletivo sobre o que se torna uma memória:

- Instruções temporárias ("corrija isso agora", "ignore isso por hoje")
- Perguntas ("qual é o prazo?")
- Conversas de depuração ("estou vendo um erro")
- Declarações vagas sem informação acionável
- Pedidos com escopo em um único PR ou tarefa

## Escopos de memória

Cada memória se aplica em um nível específico:

| Escopo | Exemplo |
|--------|---------|
| **Diretório** | "Em `src/components/ui`, sempre use nossos tokens do sistema de design" |
| **Repositório** | "Este repositório usa arquitetura hexagonal" |
| **Organização** | "Todos os repositórios usam ESLint flat config" |

## Fluxo de aprovação

Se você quiser revisar as memórias geradas por IA antes que entrem em vigor, ative **Aprovação de memórias geradas por LLM** nas configurações. As memórias entrarão em estado pendente até você aprová-las.

## Gerenciando memórias

Vá em **Configurações de Revisão de Código** → **Regras do Kody** → aba **Memórias** para visualizar, editar ou excluir memórias. Você também pode criar memórias manualmente pela interface.

Para detalhes, consulte [Regras do Kody — Memórias](/how_to_use/pt-br/code_review/configs/kody_rules#memories).
